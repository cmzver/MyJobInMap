# ADR-0001 — Фоновая очередь задач (ARQ + Redis)

## Status

Accepted

## Context

Fire-and-forget работа (push-уведомления, потенциально геокодинг, оптимизация
фото, выгрузки) выполнялась прямо в процессе web-сервера. Push, например, уходил
в `threading.Thread(daemon=True)` (`push.send_push_background`). Проблемы такого
подхода:

- поток умирает вместе с процессом → потеря задачи при рестарте/деплое;
- нет ретраев при сбое внешнего сервиса (FCM/геокодер);
- нет видимости (сколько в очереди, сколько упало);
- нагрузка делит ресурсы с обработкой HTTP-запросов.

## Decision

Ввести фоновую очередь на **ARQ + Redis** с прозрачным fallback
([app/services/task_queue.py], [app/worker.py]):

- `task_queue.enqueue(job_name, **kwargs)` — единая точка постановки.
- Режим **queue** (`TASK_QUEUE_ENABLED=true`): задача уходит в ARQ/Redis,
  выполняется отдельным процессом-воркером (`arq app.worker.WorkerSettings`) с
  ретраями (`max_tries=3`).
- Режим **inline** (по умолчанию): задача выполняется в daemon-потоке прямо в
  процессе — прежнее поведение, без зависимости от Redis. Позволяет dev/CI и
  существующим инсталляциям работать без брокера.
- **Graceful fallback**: если очередь включена, но Redis недоступен в момент
  постановки, выполняется inline — пользовательский запрос не падает.
- Одна и та же sync-реализация задачи используется в обоих режимах (в воркере —
  через `asyncio.to_thread`), поэтому поведение не расходится.

Первая переведённая на очередь задача — `push_send`. Геокодинг/оптимизация фото
добавляются тем же паттерном: загрузчик в `_JOB_LOADERS` + корутина в воркере.

## Consequences

**Плюсы**
- Ретраи и устойчивость к рестартам для push (и будущих задач).
- Изоляция фоновой нагрузки от web-процесса; масштабирование воркеров.
- Видимость через метрики (`fieldworker_queue_jobs_*`,
  `fieldworker_push_sent_total`) — см. [[Backend]] / Grafana business dashboard.
- Нулевой риск для существующих развёртываний: по умолчанию поведение прежнее.

**Минусы / компромиссы**
- Новый инфраструктурный компонент (Redis) и процесс (worker) в проде.
- Мост async-ARQ из sync-кода (выделенный event loop в потоке) — небольшая
  дополнительная сложность в `task_queue`.
- Второй писатель в БД усиливает ограничения SQLite → триггер для
  [[ADR-0002 Migration to PostgreSQL]].
