# ADR-0002 — Миграция БД с SQLite на PostgreSQL

## Status

Proposed

## Context

Прод работает на SQLite (`DATABASE_URL=sqlite:///./tasks.db`). Это было
оправдано на старте: ноль операционных затрат, один файл, простые бэкапы.
Сейчас проявляются ограничения, которые упираются именно в движок БД:

- **Конкурентная запись.** SQLite сериализует писатели (одна блокировка на
  файл). С вводом фоновой очереди ([[ADR-0001 Background task queue (ARQ)]])
  появляется второй процесс (worker), который пишет в ту же БД параллельно с
  web-процессом → растёт риск `database is locked` под нагрузкой.
- **Мультитенантность и аналитика.** Растёт число организаций и объём
  аналитических запросов (`analytics/*`, SLA). SQLite не умеет
  materialized views, partial/expression indexes ограниченно, нет нормального
  параллелизма чтения с тяжёлой записью.
- **Масштабирование.** Нельзя вынести БД на отдельный хост, поднять реплику
  для чтения/бэкапов, держать пул соединений из нескольких процессов.
- **Полнотекстовый поиск и гео.** Будущие фичи (поиск по адресам, гео-запросы,
  оптимизация маршрутов) естественно ложатся на `pg_trgm` / `PostGIS`, которых
  в SQLite нет.

Важно: инфраструктура под Postgres уже частично готова — в зависимостях есть
`psycopg2-binary`, есть `docker-compose.postgres.yml`, миграции на Alembic, а в
`config.py` — флаги `is_sqlite` / `is_postgres`. Код в основном
БД-агностичен (SQLAlchemy ORM).

## Decision

Перейти на PostgreSQL как основную БД для production и staging. SQLite
оставить только для локальной разработки и тестов (in-memory) — двойная
совместимость уже обеспечивается ORM.

### План миграции (поэтапно, обратимо)

1. **Аудит SQLite-специфики.** Найти места, завязанные на SQLite-семантику:
   - `PRAGMA`, `func.strftime`, конкатенации дат, `INSERT OR REPLACE`;
   - типы: булевы/даты, хранящиеся как TEXT/INTEGER; `JSON` колонки;
   - наивные (без tz) datetime — на Postgres использовать `timestamptz`;
   - автоинкремент и поведение `created_at`/`onupdate`.
   Закрыть несовместимости через SQLAlchemy-конструкции, нейтральные к диалекту.
2. **Параметризовать engine.** В `models/base.py` выбирать connect args и пул
   по `settings.is_postgres` (для Postgres — `pool_size`, `max_overflow`,
   `pool_pre_ping=True`; `StaticPool`/`check_same_thread` только для SQLite).
3. **Прогнать Alembic против пустой Postgres.** Убедиться, что вся история
   миграций накатывается с нуля на Postgres (а не только на SQLite). Починить
   диалект-зависимые операции (типы, server_default, индексы).
4. **CI-матрица.** Добавить в `ci-server.yml` прогон тестов против Postgres
   (services: postgres) параллельно с текущим SQLite-прогоном — чтобы
   несовместимости ловились автоматически.
5. **Скрипт переноса данных.** Одноразовый ETL прод-данных SQLite → Postgres
   (по таблицам, с переустановкой sequence для PK). Прогон на копии,
   верификация количества строк и контрольных выборок.
6. **Staging-репетиция.** Развернуть `docker-compose.postgres.yml` на staging,
   прогнать полный сценарий (web + worker + бэкапы) под нагрузкой.
7. **Cutover.** Окно обслуживания: стоп записи → финальный ETL-дельта-перенос →
   переключение `DATABASE_URL` → smoke-тест → открытие трафика.
8. **Бэкапы.** Заменить файловый бэкап SQLite на `pg_dump`/PITR
   (адаптировать `backup_service` / `backup_scheduler`).

### Откат

`DATABASE_URL` обратно на SQLite-снимок, сделанный перед cutover. Так как код
остаётся БД-агностичным, откат — это смена переменной окружения и рестарт.

## Consequences

**Плюсы**
- Реальная конкурентная запись (web + worker + будущие воркеры) без локов.
- Реплики для чтения/бэкапов, вынос БД на отдельный хост, пул соединений.
- Открывается дорога к `pg_trgm` (поиск) и `PostGIS` (гео/маршруты),
  materialized views для аналитики.

**Минусы / компромиссы**
- Появляется операционный компонент: Postgres надо мониторить, бэкапить,
  обновлять (vs один файл).
- Нужно следить за диалект-совместимостью; CI-матрица SQLite+Postgres
  удлиняет прогон.
- Разовые затраты на ETL и окно обслуживания при cutover.

**Связано с**
- [[ADR-0001 Background task queue (ARQ)]] — worker как второй писатель, главный
  технический триггер этого перехода.
