# On-demand WireGuard для домофонных панелей

Прод-сервер (`api`) должен дотянуться до панельной подсети `10.80.0.0/16`,
которая доступна только через WireGuard. Туннель поднимается **по запросу** —
только на время обращения к панели — и опускается после простоя
(`BEWARD_WG_LINGER`, по умолчанию 45 с). Логика — в
[`server/app/services/wireguard.py`](../../server/app/services/wireguard.py);
драйвер оборачивает каждое сетевое обращение в `wireguard.session()`.

## Настройка

1. **Конфиг пира.** Скопируйте пример и подставьте реальные ключи:
   ```bash
   cp deploy/wireguard/wg-intercom.conf.example deploy/wireguard/wg-intercom.conf
   ```
   Реальный `wg-intercom.conf` **в git не коммитится** (см. `.gitignore`).

2. **Креды панелей** — в `server/.env`:
   ```
   BEWARD_USER=...
   BEWARD_PASSWORD=...
   ```

3. **Запуск с overlay:**
   ```bash
   docker compose -f docker-compose.yml -f docker-compose.wireguard.yml up -d
   ```
   Overlay включает `BEWARD_WG_ENABLED=true`, добавляет `NET_ADMIN`,
   пробрасывает `/dev/net/tun` и монтирует конфиг.

## Важные моменты

- **Сужённый `AllowedIPs`.** В конфиге оставьте только панельные подсети
  (`10.80.0.0/16`, `10.125.0.0/16`, `192.168.0.1/32`). Тогда туннель не
  перехватывает остальной трафик контейнера.
- **Один пир — один хост.** Тот же пир нельзя одновременно держать на
  ноутбуке и на проде (совпадают ключ и `Address`) — будет конфликт
  хендшейков. On-demand-режим сужает окно, но в момент работы прода не
  держите туннель на ноутбуке. Для постоянной эксплуатации — отдельный пир.
- **Ротация ключей.** Ключи, которые уже где-то светились, перевыпустите.
- **root в контейнере.** `wg-quick` требует root, поэтому overlay запускает
  `api` как root. Базовый образ остаётся непривилегированным (overlay — opt-in).
- **Ядро vs userspace.** `wg-quick` использует модуль ядра `wireguard`
  (есть на большинстве хостов). Если модуля нет — поставьте userspace
  (`wireguard-go`/`boringtun`) и укажите
  `WG_QUICK_USERSPACE_IMPLEMENTATION=wireguard-go`.

## Проверка

```bash
# внутри контейнера api при включённом WG
docker compose exec api wg show
```

Либо просто нажмите «Статус замка» на панели в портале — первый запрос
после простоя поднимет туннель (+~1–2 с), последующие в пределах лингера
мгновенны.
